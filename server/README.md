# server/ — Python service (FastAPI + SQLAlchemy 2 + Alembic)

Postgres-only (dev, test, and prod — no SQLite anywhere), psycopg v3, MinIO
for S3-compatible object storage, Docker Compose for local dev.

## Sync, not async — and why

SQLAlchemy runs in **synchronous** mode (`create_engine` + `Session` on
psycopg v3). FastAPI executes sync endpoints in a threadpool, so nothing
blocks the event loop; Alembic, pytest fixtures, and transaction semantics
stay simple; and because psycopg v3 has a native async side, a later switch is
localized to `app/db.py` and endpoint signatures rather than being a rewrite.
Revisit if the service becomes fan-out I/O heavy (many concurrent S3/DB calls
per request).

## Layout

```
server/
  app/
    main.py       app factory + /health (does a real SELECT 1)
    config.py     pydantic-settings (DATABASE_URL, S3_* / MinIO)
    db.py         engine/session + get_db dependency
    models.py     Base (naming conventions), smoke_notes, replays
    ttr.py        .ttr parsing/validation, ported from src/replay/parse.ts
    replays.py    POST /replays (multipart ingest)
    storage.py    boto3 S3 wrapper (bucket created by compose minio-init)
  alembic/        env.py resolves URL from attributes or DATABASE_URL
  tests/          migrations-once + rollback-per-test fixtures
```

## POST /replays

Multipart upload (`file` = the `.ttr`, optional `uploader` form field).
Solo `.ttr` only — multiplayer `.ttrm` is rejected with **415**
("multiplayer replays are not supported"), detected both by extension
(before the body is read) and by shape (`replay.rounds` /
`replay.leaderboard`, right after JSON parse). Size is capped at **3 MB**
(413); unparseable files get **422**.

Accepted files are SHA-256 hashed, deduped by hash (**201** new, **200**
existing with the same body), stored at `replays/<sha256>.ttr` in the bucket,
and recorded in the `replays` table with the same metadata the frontend
extracts (`player_username`, gamemode, seed, bagtype, pps/apm/vsscore,
frames, piece_count) plus `uploader`. The reconstructable allowlist mirrors
`src/engine/adapter.ts` (7-bag, 14-bag, classic, pairs, total mayhem,
7+1/7+2/7+x-bag, standard 10×20 board); **Zenith is stored parse-only**
(`reconstructable = false`) since the server has no engine to drift-cap it.

## Workflow (from the repo root)

```
make up                      # build + start db, db-test, minio (+bucket), api
curl localhost:8000/health   # {"status":"ok","database":"ok"}
make revision m="add replays"   # alembic autogenerate inside the api container
make migrate                 # alembic upgrade head against dev Postgres
make test                    # pytest against the tmpfs db-test on :5433
make logs / make down
```

The api container bind-mounts `server/` and runs `uvicorn --reload`, so code
and newly autogenerated migration files appear on the host immediately. On
startup it applies `alembic upgrade head` before serving.

## Test strategy

`TEST_DATABASE_URL` points at the throwaway `db-test` container (tmpfs — data
dies with the container, which is desired). The session-scoped `engine`
fixture runs the real migrations once; each test then gets a `Session` bound
to a connection-level transaction with `join_transaction_mode=
"create_savepoint"`, so code under test can `commit()` (savepoint release)
while the whole test rolls back at teardown. Fast, isolated, migration-true.

## Environment

Copy `.env.example` → `.env` (git-ignored). In-network hostnames (`db`,
`db-test`, `minio`) are for containers; from the host use `localhost:5432`,
`localhost:5433`, `localhost:9000` (MinIO console on `:9001`).
